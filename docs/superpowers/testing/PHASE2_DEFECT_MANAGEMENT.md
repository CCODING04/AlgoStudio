## 1. 缺陷管理概述

### 1.1 目标

- 确保所有缺陷被正确记录、跟踪和解决
- 提供透明的缺陷状态视图
- 支持质量指标分析

### 1.2 缺陷生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                      缺陷生命周期                             │
│                                                             │
│  [发现] ──→ [记录] ──→ [分配] ──→ [修复] ──→ [验证] ──→ [关闭]│
│     │                                          │            │
│     │                                          ↓            │
│     │                                    [重新打开]          │
│     │                                          │            │
│     └──────────────────────────────────────────┘            │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 状态定义

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| New | 新发现缺陷 | 测试/用户报告 |
| Open | 已确认待修复 | QA 确认 |
| In Progress | 修复中 | 开发人员开始修复 |
| Fixed | 已修复待验证 | 开发人员提交修复 |
| Verified | 验证通过 | QA 验证通过 |
| Closed | 已关闭 | 无异议 |
| Reopened | 重新打开 | 验证失败 |
| Won't Fix | 不修复 | 确认不修复 |
| Duplicate | 重复缺陷 | 确认重复 |

## 3. 缺陷严重程度和优先级

### 3.1 严重程度定义

| 等级 | 定义 | 示例 |
|------|------|------|
| Critical | 系统崩溃、数据丢失 | API 宕机、任务数据丢失 |
| High | 核心功能不可用 | 无法创建任务、无法部署 |
| Medium | 功能部分受损 | SSE 延迟高、部分页面异常 |
| Low | 轻微问题 | UI 显示不准确、文档错误 |

### 3.2 优先级定义

| 优先级 | 定义 | 修复时限 |
|--------|------|----------|
| P0 | 必须立即修复 | 24h |
| P1 | 高优先级 | 3d |
| P2 | 正常优先级 | 1w |
| P3 | 低优先级 | 发布后 |

### 3.3 严重程度和优先级映射

| 严重程度 | 默认优先级 | 说明 |
|----------|------------|------|
| Critical | P0 | 立即处理 |
| High | P1 | 尽快修复 |
| Medium | P2 | 正常流程 |
| Low | P3 | 视情况修复 |

## 4. 缺陷跟踪流程

### 4.1 缺陷发现

1. 测试执行中发现缺陷
2. 记录缺陷到 `docs/superpowers/testing/defects/BUG-{ID}.md`
3. 更新缺陷汇总表

### 4.2 缺陷确认

1. QA 确认缺陷有效
2. 设置严重程度和优先级
3. 分配给对应的开发人员

### 4.3 缺陷修复

1. 开发人员接收缺陷
2. 分析并修复
3. 在缺陷报告中填写修复信息
4. 提交代码修复

### 4.4 缺陷验证

1. QA 拉取修复代码
2. 在相同环境中复现测试
3. 验证缺陷是否修复
4. 更新缺陷状态

### 4.5 缺陷关闭

1. 验证通过后关闭缺陷
2. 更新缺陷汇总表
3. 更新测试报告

## 6. JIRA 集成 (可选)

### 6.1 JIRA 配置 (可选)

如果项目使用 JIRA，可以配置集成：

```yaml
# jira-config.yml
jira:
  enabled: false  # 设置为 true 启用
  url: "https://your-company.atlassian.net"
  project_key: "ALGO"
  api_token: "${JIRA_API_TOKEN}"  # 从环境变量读取
```

### 6.2 JIRA 字段映射

| 本地字段 | JIRA 字段 |
|----------|-----------|
| 标题 | Summary |
| 描述 | Description |
| 严重程度 | Severity |
| 优先级 | Priority |
| 状态 | Status |
| 环境 | Environment |

### 6.3 本地优先策略

**重要:** 本地缺陷管理优先于 JIRA

- 所有缺陷必须先记录到本地文件
- JIRA 仅作为可选同步目标
- 保持本地文件为唯一真相来源

## 7. 质量指标

### 7.1 缺陷指标

| 指标 | 计算方式 | 目标 |
|------|----------|------|
| 缺陷密度 | 缺陷数/代码行数 | < 5/1000 行 |
| P0 缺陷数 | 统计 P0 缺陷 | 0 |
| 平均修复时间 | 总修复时间/缺陷数 | P0 < 24h, P1 < 3d |
| 缺陷逃逸率 | 生产缺陷数/总缺陷数 | < 5% |

### 7.2 测试覆盖指标

| 指标 | 计算方式 | 目标 |
|------|----------|------|
| E2E 覆盖率 | 已覆盖场景/总场景 | >= 90% |
| 压力测试覆盖 | 已覆盖场景/总场景 | >= 80% |
| UAT 通过率 | 通过用例/总用例 | 100% |

## 10. 附录

### 10.1 常见缺陷分类

| 分类 | 说明 | 示例 |
|------|------|------|
| 功能缺陷 | 功能不工作 | 任务创建失败 |
| 性能缺陷 | 性能不达标 | API p95 > 100ms |
| 界面缺陷 | UI 问题 | 页面显示异常 |
| 安全缺陷 | 安全问题 | 未授权访问 |
| 兼容缺陷 | 兼容性问题 | 浏览器不兼容 |

### 10.2 缺陷关闭标准

缺陷可关闭当且仅当：
1. 所有复现步骤已验证通过
2. 相关自动化测试通过
3. Code Review 通过
4. QA 确认
